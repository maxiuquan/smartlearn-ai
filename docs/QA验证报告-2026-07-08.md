### 【2. 纯逻辑隔离单测】（33/33 ✅）

> 因 httpx / pydantic_settings 未安装，SSRF 与 JWT 测试为**复制函数副本 + stub** 的等价测试（非 import 真实模块）；`/restore` 为复制正则 + 复刻 resolve 越界逻辑；vocab 为静态确认。测试脚本位于 `C:\tmp\qa_smartlearn\unit_tests.py`，可复跑。

- **2.1 SSRF `is_safe_url`（16 项）**：复制 `app/security/ssrf.py` 纯逻辑，`_resolve_hostname` 用可控 DNS 桩替代。
  - 拦截：localhost、169.254.169.254、10.x、192.168.x、127.x、file://、ftp://、gopher://、高危端口 6379、DNS 重绑定（解析到内网）、双栈含私网。
  - 放行：公网域名解析公网 IP、白名单精确/后缀匹配、非字符串/空 URL 被拒。
- **2.2 `validate_jwt_secret`（7 项）**：复制 `app/core/config.py` 函数体 + `KNOWN_WEAK_JWT_SECRETS`。
  - 拒：空、None、`change-me-in-production`、`your-jwt-secret-please-generate-with-openssl-rand-hex-32`、长度<32。
  - 放：长度≥32 强密钥、64 位 hex。
- **2.3 `/restore` backup_id（8 项）**：复制正则 `^[A-Za-z0-9_.-]+$` + 复刻 `system.py` 两层护栏。
  - 合法 `abc.sql.gz` / `backup_2024.sql.gz` / 含下划线连字符 通过。
  - `/`、`\`、绝对路径 `/etc/passwd` 被正则拦截；`..`、`../../etc/passwd` 被**第二层 resolve 越界判定**拦截。
  - ⚠️ **重要细节**：正则本身会放行 `..`（点号在字符集内），实际依赖第二层 `resolve()` 越界判定兜底——纵深防御成立，但单看正则并不够。已记录。
- **2.4 vocab `case` 用法**：`grep` 确认 `vocab.py` 第 108 行 `from sqlalchemy import ... case`，第 129/132/135 行调用 `case((uwp.c.status == "mastered", 1), else_=0)` 形式的**顶层 `case`**，全文件无 `func.case`（B5 修复确认）。

### 【3. 安全不变量核查】

- **3.1 无裸 DDL（✅）**：`grep` 全 scripts 目录，`CREATE TABLE`/`DROP TABLE` 仅出现在各文件第 8 行模块 docstring（描述「已删除」）。运行代码均为 ORM：`import_vocabulary.py` 用 `insert(...).on_conflict_do_update`（引用 `stmt.excluded`）+ `select(VocabularyWord)`；`seed.py`/`import_questions.py`/`import_knowledge.py` 的 `db.execute` 均为 ORM upsert/select。**B1/B2 确认**。
- **3.2 受保护路由鉴权（✅）**：
  - `chat_router.py:48` `/chat` → `require_auth`
  - `media_router.py:64/92/131` `/media/tts|stt|image` → `require_auth`；`/stt` 额外 `is_safe_url` + `safe_http_get_bytes` SSRF 防护（C2 接线确认，`media_router.py:102-115`）
  - `moderation_router.py:71/90` `/moderation` → `require_auth`
  - `prompt_router.py:241` `POST /prompts/{id}` → `require_service_or_admin`（C3 越权写防护确认）
  - `word_games_router.py:25/38/65` `start_game|submit_answer|get_leaderboard` → `require_auth`
- **3.3 provider 异常不再静默 mock（⚠️ 见第 6 节）**：
  - ✅ `moderation`（`router.py:590-604`）：异常 → `flagged=True`（fail-closed）；供应商未配置 → `flagged=True`。
  - ✅ `embedding`（`router.py:347-359,384-396`）：异常 → 抛 `ProviderUnavailableError`，**绝不返回假向量**。
  - ✅ `chat`（`openai_compat.py:117-126,168-177`）：异常 → raise，由 AIRouter 区分离线/故障。
  - ❌ **`cogview` 图像生成（`cogview.py:99-102`）：`except Exception: return self._mock_image(prompt)`** —— 在线调用失败时**静默返回灰色占位 SVG**（HTTP 200），这是 C4「异常不再静默 mock」的残留，且不受 fail-closed 保护。属非安全关键路径，但偏离整改目标。
  - ⚠️ 观察：`_mock_*` 系列方法（embedding/chat/image/tts/stt/moderate）仍保留在代码中，仅在 `offline_mode`（无 API Key）或 `client is None` 时可达；经核对 `_get_client()` 仅在 `offline_mode=True` 时返回 None，故在线配置 Key 的场景不会触发静默 mock。功能上生产安全不受损，但与「不再有 mock fallback」的措辞不完全一致。
  - ⚠️ 观察：`moderation.py:112-120` 在线模式下若 LLM 返回非 JSON，构造默认 `flagged:False` 结果——属于「调用成功但解析失败」的轻微 fail-open 边界，非异常路径。
- **3.4 弱密钥 + 启动守卫（✅）**：
  - `services/api/app/core/config.py:16-25` `KNOWN_WEAK_JWT_SECRETS` **含 `change-me-in-production`**（及 change-me / your-... 等）。
  - `services/api/app/main.py:38-50` `_startup_security_check` 调用 `validate_jwt_secret()`；若抛 `ValueError` 且 `settings.is_production` → `sys.exit(1)`（B6 production fail-fast 确认）。
  - `services/ai-engine/app/auth.py:35-69` `_WEAK_JWT_SECRETS` 含 `change-me-in-production`；`app/main.py:31` 生命周期 `ensure_auth_configured()` 在鉴权开启但缺密钥时抛 `RuntimeError` 阻止启动（C1 fail-fast 确认）。
- **3.5 `.gitignore`（✅）**：`F:\xiangmu\xuexi\.gitignore:129-130` 含 `.workbuddy/` 与 `**/.workbuddy/`。
