# SmartLearn AI 增量整改 PRD

> 文档版本: v1.0 ｜ 日期: 2026-07-08 ｜ 作者: 许清楚（产品经理）
> 依据: 《全量代码审查报告-2026-07-08.md》（权威事实来源，约 293 源码文件 + 21 数据文件逐文件审查）
> 目标: 将项目从「健康度 4/10、按文档无法部署」推进到「可部署、基本安全、文档自洽」的理想基线状态
> 范围声明: 本轮为**增量整改**，**不新增业务功能**，仅做修复 / 对齐 / 补全结构。真实大规模考研真题内容不编造，仅定义 schema + 代表性样例，并在文档标注「真实大规模数据需后续提供」。

---

## 1. 产品目标

本轮整改后系统应达到以下可验证状态：

| 目标 | 可验证定义 |
|---|---|
| **G1 可部署** | 按文档 `cp .env.example .env` + 填密钥后 `make up` 能一次性拉起 db/redis/api/ai-engine/admin/student-web/nginx 全栈，无构建失败、无启动即崩；健康检查全绿 |
| **G2 基本安全** | 生产环境强制强 JWT 密钥（弱/默认密钥 fail-fast 拒绝启动）；ai-engine 全部写/敏感接口加全局鉴权；消除 SSRF 与越权写；不再静默伪装供应商故障为成功 |
| **G3 文档自洽** | README/文档声明的技术栈、前端清单、数据规模与代码真实状态一致；空壳模块明确标注「规划中」 |
| **G4 数据可灌** | `make migrate && make seed && make import-all` 可成功写入数据库，脚本复用 ORM/Alembic，无自建孤儿表、无路径错误 |
| **G5 功能不空心** | 真题/习题册具备正确 schema + 代表性样例（可端到端走通）；词汇规模声明对齐真实值 |

正交性检查：G1(部署)、G2(安全)、G3(文档)、G4(数据)、G5(内容) 互不重叠。

## 2. 用户故事

- **US1（部署者）**：作为首次部署者，我希望复制 `.env.example` 并填好密钥后执行 `make up` 能一次性启动全栈，这样我无需逐文件排查构建/配置矛盾。
- **US2（终端用户/学生）**：作为学生用户，我能在 student-web 上登录、刷新任意页面不 404、调用后端 API 正常答题，这样学习流程不被打断。
- **US3（运维/安全）**：作为运维，我希望生产环境在密钥缺失/过弱或 ai-engine 未鉴权时拒绝启动，这样不会把"看似正常但投喂假答案/可被匿名调用"的服务暴露上线。
- **US4（内容运营）**：作为运营，我希望真题/习题册/词汇有正确的数据结构和样例，我能按 schema 批量灌入真实考题，这样产品功能可真正使用而非空壳。
- **US5（后续开发者）**：作为新接手开发者，我希望 README 与实际技术栈、模块清单、数据规模一致，这样不会按错误文档做开发决策。

## 3. 需求池

### P0（必须 · 阻断级）

#### P0-1：移除空壳模块 web / mobile-web 出生产编排
- **问题来源**：审查报告 3.4、四-P0-2。apps/web、apps/mobile-web 各仅含 `Dockerfile` + `package.json`，**零源码**（已目录列举确认）；`Dockerfile.web/mobile` 用 `npm ci` 无 lock、无 `src/`，mobile-web 还缺被 COPY 的 `nginx.conf`；二者构建失败，且 `nginx.depends_on` 含 web/mobile-web，**阻断整个 compose 栈启动**。
- **整改动作（采用"移除方案"）**：
  1. 从 `docker-compose.yml` 删除 `web`、`mobile-web` 两个 service 及其在 `nginx.depends_on` 中的引用。
  2. 从 `infra/nginx/conf.d` 移除指向 web/mobile-web 的 server 配置（或保留注释占位）。
  3. 将 apps/web、apps/mobile-web 目录**删除**（git 历史保留），并在根 README 前端清单注明「web/mobile-web 规划中，暂未实现」；或保留目录并放置 README 标注「规划中」——见待确认问题①。
  4. 更新文档前端清单，使其与真实存在的前端（admin / mobile / student-web）一致。
- **验收标准**：
  - `docker compose config` 通过，且 `web`/`mobile-web` 不再出现在 service 列表。
  - `make build` 不再因 web/mobile-web 构建失败而中断；`make up` 全栈可起（不计这俩服务）。
  - 根 README 前端清单不含 web/mobile-web 的"已实现"描述；若保留目录，其 README 明确标注"规划中"。
- **决策说明**：选移除而非补最小实现——补空应用仍是无功能占位且增加维护面，而移除能立即解除对整栈启动的阻断，符合"先让系统能部署"的优先序。

#### P0-2：修正 admin 依赖声明 + 提供 nginx.conf
- **问题来源**：审查报告 3.1、四-P0-1。admin/package.json 写 Vue/Element/Pinia、build 脚本 `vue-tsc -b`，但实际是 **React 18 + AntD 5**（已确认源码）。Dockerfile.admin 第 28 行 `COPY apps/admin/nginx.conf` 不存在 → 容器构建中断；且缺 `package-lock.json`，`npm ci` 失败（已确认无 lock 文件）。
- **整改动作**：
  1. 将 package.json 依赖修正为真实 React 栈：dependencies 改为 `react`/`react-dom`/`antd`/`@ant-design/pro-components`/`zustand`/`axios`/`echarts` 等；devDependencies 改为 `vite`/`@vitejs/plugin-react`/`typescript`/`eslint` 等；build 脚本改为 `vite build`（去掉 vue-tsc）；description 改为 React 后台。
  2. 新增 `apps/admin/nginx.conf`（SPA 配置：root `/usr/share/nginx/html`、`try_files` 兜底 `index.html`、API 反向代理 `/api` → `api:8000`）。
  3. 生成并提交 `apps/admin/package-lock.json`，或把 Dockerfile 的 `npm ci` 改为 `npm install`。
- **验收标准**：
  - `npm install` 成功，依赖树为 React 栈；`npm run build` 产出 `dist/`。
  - `docker build -f apps/admin/Dockerfile.admin .` 成功（nginx.conf 存在、npm 安装成功）。
  - package.json 不再出现 `vue`/`element-plus`/`pinia`/`vue-tsc`。

#### P0-3：student-web 生产 Docker 加反向代理 + SPA fallback
- **问题来源**：审查报告 3.3、四-P0-8。Dockerfile.student 无 nginx 反向代理，`client.baseURL` 默认空串（dev 代理不进生产），BrowserRouter 深链接无 fallback → API 不可达 + 刷新 404。
- **整改动作**：
  1. 新增 `apps/student-web/nginx.conf`：配置 API 反向代理（如 `/api` → `api:8000`）、SPA `try_files $uri /index.html` 兜底。
  2. Dockerfile.student 多阶段：先 `vite build`，再用 nginx 承载，COPY nginx.conf 到 conf.d。
  3. 前端以构建期环境变量 / 运行时注入方式设置 API baseURL（不使用空串默认）。
- **验收标准**：
  - 容器启动后，浏览器访问任意深链接路径刷新均返回 index.html（不 404）。
  - 前端能经 nginx 代理成功调用 api（`/health` 可达）。

#### P0-4：mobile package.json 修正 main/entry + 补依赖
- **问题来源**：审查报告 3.2、四-P0-9。package.json `"main": "expo-router/entry"` 与 app.json 启用 expo-router 但**无 `app/` 目录**、实际入口 `App.tsx` 冲突（已确认）；`AchievementPopup` 导入不存在的 `lottie-react-native`。
- **整改动作**：
  1. 统一入口：若采用 expo-router，则建立 `app/` 目录与路由结构；否则移除 expo-router 依赖与 app.json 插件，将 `"main"` 指向 `node_modules/expo/AppEntry.js` 或 `./App.tsx`（二选一并保持一致）。
  2. 安装缺失依赖 `lottie-react-native`（及 `lottie-ios` 如适用），或移除对该库的导入（若成就弹窗暂不使用动画则先移除导入以解除崩溃）。
- **验收标准**：
  - `npm install` 无缺失依赖报错；`expo start` 能正常加载入口，无 "main entry conflict" / "module not found" 崩溃。
  - 工程可进入 Expo 开发服务器（健康度自检）。

#### P0-5：import_*.py 改用现有 ORM/metadata，停止自建表
- **问题来源**：审查报告 3.5、四-P0-4。services/api/scripts/import_*.py 用 `CREATE TABLE IF NOT EXISTS questions(...)` 等裸 SQL 自建表（已抽样 import_questions.py 确认），列定义与 SQLAlchemy ORM/Alembic 真实 schema 脱节，在已迁移库上运行直接报错。
- **整改动作**：
  1. 重写导入脚本：复用 API 服务的 SQLAlchemy ORM 模型（import 现有 models）与 session，通过 `bulk_insert_mappings` / ORM 对象写入，删除全部 `CREATE TABLE` 自建逻辑。
  2. 数据目录解析沿用现有 `data/` 结构；字段映射对齐 ORM 列名。
  3. 脚本接收 `DATABASE_URL`；与 Alembic 迁移后的库兼容（先 migrate 后 import）。
- **验收标准**：
  - 在 `make migrate` 之后的库上运行 `make import-all` 成功，无 SQL 冲突报错。
  - 导入后各表数量与 `data/*.json` 一致（数学 5026、英语 3224 等真实值）。
  - 脚本中不再出现 `CREATE TABLE`。

#### P0-6：seed.py 孤儿 achievements 表移除/对齐
- **问题来源**：审查报告 3.5、四-P0-5。services/api/scripts/seed.py 用裸 `CREATE TABLE IF NOT EXISTS achievements(...)` 自建表（已确认），ORM 未定义、Alembic 未管理；同时 Makefile `seed` 调用 `scripts/seed.py`（容器 `/app/scripts/`），路径不一致（见 P0-12）。
- **整改动作**：
  1. 若成就系统为计划功能：在 ORM 中正式定义 `Achievement` 模型并经 Alembic 迁移管理，seed 通过 ORM 写入合法数据。
  2. 若本轮不做成就系统：删除 seed.py 中 achievements 自建表/写入逻辑，确保 seed 仅写入与 ORM 一致的初始数据（用户/角色/基础配置等）。
  3. 确保 Makefile 实际调用的路径存在可执行的 seed 入口（见 P0-12）。
- **验收标准**：
  - 全库无"ORM 未定义但物理存在"的表；`\dt` 中 achievements（若保留）有对应 ORM 与迁移版本。
  - `make seed` 可成功执行且不报错。

#### P0-7：默认 JWT_SECRET 生产不可伪造
- **问题来源**：审查报告 3.5、五-P1 安全。默认 `JWT_SECRET="change-me-in-production"` 能通过启动校验，攻击者可伪造 token。
- **整改动作**：
  1. API 启动时校验 `JWT_SECRET`：长度 ≥ 32、不得等于默认弱值、生产环境（`ENVIRONMENT=production`）必须显式设置；校验失败则 fail-fast 退出（非零退出码）。
  2. `.env.example` 中 JWT_SECRET 仅作占位提示，文档明确"生产必须替换为随机密钥"。
- **验收标准**：
  - 使用默认/弱密钥启动生产 → 进程启动失败并打印明确错误。
  - 使用 `openssl rand -hex 32` 生成的密钥 → 正常启动，签发 token 可被正确校验。

#### P0-8：ai-engine 全站加全局鉴权
- **问题来源**：审查报告 3.6、四-P0-10。/chat、/moderation、/media/*、/prompts 写接口均匿名可达；main.py 无任何 auth 中间件/依赖（已确认 main.py 仅注册路由、无鉴权）。
- **整改动作**：
  1. 在 ai-engine 引入统一鉴权依赖（API Key 或共享 JWT/网关转发鉴权）：对 `/chat`、`/moderation`、`/media/*`、`POST /prompts`、`/word-games` 等写/敏感接口强制校验；`/health`、`/docs` 可豁免。
  2. 通过环境变量（如 `AI_ENGINE_API_KEY` / `AI_ENGINE_AUTH_ENABLED`）控制，与 API 网关共享密钥；compose 注入该密钥。
  3. 未携带合法凭证的请求返回 401/403。
- **验收标准**：
  - 匿名调用 `POST /chat`、`POST /media/stt`、`POST /prompts` → 401/403。
  - 携带合法密钥（由网关/前端注入）→ 200。
  - 自动化测试覆盖鉴权拦截。

#### P0-9：ai-engine /media/stt SSRF 限制
- **问题来源**：审查报告 3.6、四-P0-10。/media/stt 服务端代抓任意 URL，可触内网/云元数据。
- **整改动作**：
  1. 限制可访问地址：仅允许 http/https、解析后 IP 不属私有/环回/链路本地/云元数据段（如 169.254.169.254）、禁止重定向到内网、可配置允许域名白名单。
  2. 设置请求超时与响应大小上限；禁用跟随跨网段重定向。
- **验收标准**：
  - 请求内网/元数据地址（如 `http://169.254.169.254/`、`http://localhost/`）→ 被拒绝。
  - 仅允许的公网音频 URL 可正常处理；超时与大小限制生效。

#### P0-10：ai-engine POST /prompts/{id} 越权写
- **问题来源**：审查报告 3.6、四-P0-10。POST /prompts/{id} 无鉴权可注入恶意 prompt 污染对话。
- **整改动作**：
  1. 在 P0-8 鉴权基础上，增加归属/权限校验：仅 prompt 所有者或管理员可修改；记录操作审计。
  2. 校验目标 prompt 存在且属于当前调用方；越权返回 403/404（不泄露存在性）。
- **验收标准**：
  - A 用户修改 B 用户的 prompt → 403。
  - 所有者修改自己的 prompt → 成功，审计日志有记录。

#### P0-11：ai-engine 异常不再静默伪装成功
- **问题来源**：审查报告 3.6、四-P0-11。所有 provider 异常被静默替换为"模拟响应"，线上故障/限流长期投喂假答案且不可观测。
- **整改动作**：
  1. 区分"显式离线降级"（配置 offline_mode 或所有 provider 未配置 key）与"运行时故障"：
     - 离线模式：可返回模拟响应，但需在响应体/日志明确标注 `simulated: true` 与原因。
     - 运行时异常（供应商 5xx/限流/超时）：向上抛出明确错误（HTTP 502/503），日志记录 provider、错误类型、trace_id；**不得**静默替换为成功响应。
  2. 暴露可观测指标/日志（错误计数、失败 provider），供运维告警。
- **验收标准**：
  - 供应商返回 500/429 → 接口返回错误（非 200 假答案），日志含明确错误。
  - 离线模式下的模拟响应对调用方透明标注，不伪装为真实 AI 输出。
  - 单元/集成测试覆盖"故障显式报错"分支。

#### P0-12：Makefile 调用的脚本缺失 → 补齐或修正路径
- **问题来源**：审查报告 3.8、四-P0-5。Makefile 的 `seed/import-kp/import-q/import-vocab/import-all` 调用 `scripts/seed.py`、`scripts/import_*.py`、`scripts/import_all.py`，但仓库根 `scripts/` 下无 .py（实际导入/种子脚本在 `services/api/scripts/`），且路径不一致 → 数据初始化完全失败。
- **整改动作**：
  1. 路径对齐：将 Makefile 中 `python scripts/import_*.py` 修正为 `python services/api/scripts/import_*.py`（与现有脚本位置一致），或调整 Dockerfile 把脚本复制到 `/app/scripts/`。
  2. 确保 `services/api/scripts/seed.py` 在 Makefile 调用路径下可达（P0-6 已处理其自建表问题）。
  3. 验证 `make migrate && make seed && make import-all` 全链路可跑通。
- **验收标准**：
  - `make seed`、`make import-kp/q/vocab/all` 均成功执行，无 "No such file" 报错。
  - 数据库出现预期初始数据。

#### P0-13（补充部署阻断）：compose 强制 MINIO_* + .env.example 缺密码
- **问题来源**：审查报告 3.8、四-P0-6。docker-compose.yml 中 api 服务 `MINIO_ACCESS_KEY:?err` / `MINIO_SECRET_KEY:?err` 强制必填（但 MinIO 已注释禁用）；`.env.example` 缺 `POSTGRES_PASSWORD`、`REDIS_PASSWORD` → `make up` 直接退出。
- **整改动作**：
  1. 移除 api/ai-engine 中对已禁用 MinIO 的强制 `:?err` 要求（改为 `:-` 默认值或仅在使用 MinIO 时校验）。
  2. `.env.example` 补全 `POSTGRES_PASSWORD`、`REDIS_PASSWORD`（占位提示必填），或 compose 改为非强制；`make up` 不再因缺变量退出。
- **验收标准**：
  - 仅填文档"最低必填项 + 密码"后 `make up` 完整启动，无 `err: ... 必须设置` 中断。

#### P0-14（补充部署阻断）：nginx 443 引用缺省证书导致容器启动即崩
- **问题来源**：审查报告 3.8、四-P0-7。infra/nginx 443 server 引用缺省 ssl 证书，容器启动失败；`/nginx-health` 仅定义于 443，80 健康检查永远失败（已确认仅 `infra/nginx/nginx.conf` 存在，无证书）。
- **整改动作**：
  1. 让 nginx 无证书也能启动：将 443 server 改为"有证书才 include"的条件配置（如证书文件存在才 listen 443），或提供自签名占位证书（开发用）。
  2. 在 80 server 内定义 `/nginx-health` 端点（返回 200），使 80 健康检查可达。
- **验收标准**：
  - 无证书情况下 `make up` 后 nginx 容器保持健康（healthcheck 绿）。
  - `curl http://localhost/nginx-health` 返回 200。
