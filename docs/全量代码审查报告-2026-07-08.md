# SmartLearn AI 全量代码审查报告

> 审查日期：2026-07-08
> 审查范围：逐功能 / 逐模块 / 逐文件，不遗漏
> 审查方式：8 个独立审查代理并行覆盖各边界（admin / mobile / student-web / web+mobile-web / api / ai-engine / data / infra+scripts+docs），主理人汇总
> 真实源码覆盖：前端 5 个应用、后端 2 个服务、数据 21 个 JSON、脚本 14 个、基础设施 6 个配置 + compose/Makefile/文档，合计 **约 293 个源码文件 + 21 个数据文件全部逐文件审查**

## 一、执行摘要（TL;DR）

SmartLearn AI 是一个**功能设想宏大、但工程实现与文档严重脱节、且当前无法按文档一键部署**的项目。代码主体质量参差：后端 API 管理端与 AI 抽象层设计有亮点，但**前端工程配置普遍错误到无法构建**、**部署编排存在多个阻断级矛盾**、**AI 引擎安全与可用性缺陷高危**。数据层引用完整性良好，但规模声明严重虚标、真题/习题册为空壳。

**总体健康度：4 / 10**（不可用状态，需先解决一批 P0 阻断）。

一句话结论：**文档描述的是一个"理想中已完成的平台"，代码是一份"半成品 + 大量脚手架占位"，二者不能互相印证。**

## 三、逐模块审查结论

### 3.1 apps/admin（React+AntD 管理后台）— 健康度 3/10
- 实际技术栈 React 18 + AntD 5 + ProComponents + Zustand + ECharts，但 `package.json` 写成了 Vue/Element/Pinia，`build` 脚本是 `vue-tsc -b`。
- **P0**：① package.json 依赖与构建脚本完全错误，无法 `npm install`/`vite build`；② `Dockerfile.admin` 引用不存在的 `apps/admin/nginx.conf`，容器构建中断。
- **P1**：多处裸 `fetch` 不带 Authorization（exportUsers/exportQuestions/exportWords/exportStatisticsReport/WorkbookDetail 题目加载）→ 401 失效；QuestionList "查看"路由 404；QuestionEdit 选项以 TextArea 存字符串而非 `{key,content}[]`；SystemSettings 两个 Form 共用同一 `useForm` 冲突；refreshToken 已实现从未调用；严格 tsconfig 下多处未用导入编译失败。
- **P2**：死按钮（批量导入/导出）、死代码、token 存 localStorage（XSS 风险）、Chart 全量 import echarts 包体过大。
- 亮点：目录分层清晰、Axios 拦截器设计合理、CRUD 模式一致、列表批量操作有二次确认。

### 3.2 apps/mobile（React Native）— 健康度 5/10
- **P0**：① `package.json` 的 `"main": "expo-router/entry"` 与 `app.json` 启用的 expo-router 插件冲突（无 `app/` 目录、实际入口 App.tsx）→ 启动崩溃；② `AchievementPopup` 导入 `package.json` 中不存在的 `lottie-react-native` → 模块加载崩溃。
- **P1**：`checkAuth` 已实现但从未调用 → 登录态不恢复；`getCurrentPlan/getTodayTasks` 与 `setCurrentPlan/setTodayTasks` 从未调用 → 首页进度/倒计时恒空；LearnScreen 构造题型筛选却未传给 `getRandomQuestions`；首题 `timeSpent` 为天文数字（startTime 初始 0）；计算/证明题"查看答案"提交空串恒判错且 DrawingBoard 无法绘制（TouchableWithoutFeedback 无 move 事件）；PlanScreen（425 行）未注册导航器 → 死代码。
- **P2**：大量写死 mock 数据、空 `onPress` 死链、apiClient 响应契约歧义、答案/进度不落库、无全局错误边界。
- 亮点：token 用 `expo-secure-store`（非明文）、Skia 图形实现质量好、类型定义完备、列表多用 FlatList。

### 3.3 apps/student-web（React+Vite）— 健康度 6/10
- **P0**：Docker 部署下 API 不可达 + SPA 刷新 404——`Dockerfile.student` 无 nginx 反向代理，`client.baseURL` 默认空串（dev 代理不进生产），BrowserRouter 深链接无 fallback。
- **P1**：25 款游戏实为 3 类通用题型+数据声明；`startGame` 未传 `game_id`（同类游戏内容一致）；**无用户身份认证**（写死 `student_001`）；三个答题页 + 两个题卡近乎逐行重复。
- **P2**：跨科目游戏用 QuestionCard 而非 MathQuestionCard（公式不渲染）；计时器依赖 timeLeft 反复重建 interval；无 AbortController；"再来一局"跳回大厅语义不符。
- **修正说明**：此前盘点曾误判"dist/ 被提交仓库"，经 git 验证 `**/dist/` 已被 .gitignore 正确忽略，**未进入版本库**，特此更正。
- 亮点：`FormulaText` XSS 防护到位（先转义再注入 KaTeX）、组件职责清晰、空/错/加载态齐全。

### 3.4 apps/web 与 apps/mobile-web（空壳）— P0 部署阻断
- 两个目录各仅 2 个文件（Dockerfile + package.json），**零源码**。
- `Dockerfile.web`/`Dockerfile.mobile` 用 `npm ci` 但无 `package-lock.json`、无 `src/`、mobile-web 还缺被 `COPY` 的 `nginx.conf` → 构建最早阶段即失败。
- `docker-compose.yml` 的 `nginx` 服务 `depends_on` web/mobile-web，二者构建失败会**阻断整个栈启动**。
- 结论：确属"架构文档/编排配置与实际代码严重不符"，且比"构建出空应用"更糟——**构建即失败**。

### 3.5 services/api（FastAPI 后端）— 健康度 5/10
- **框架确认**：100% FastAPI + SQLAlchemy 2.0(async) + Alembic + Celery，**无 NestJS/Prisma/TS**。README 框架描述完全错误。
- **P0**：① 数据导入脚本（`import_knowledge/import_questions/import_vocabulary/seed`）用 `CREATE TABLE` 自建表，列定义与 ORM/Alembic 真实 schema 严重脱节，在已迁移库上运行直接报错；② `seed.py` 建 ORM 未定义、迁移未管理的 `achievements` 孤儿表。
- **P1**：默认 `JWT_SECRET="change-me-in-production"` 能绕过启动校验（可伪造 token）；`/system/restore/{id}` 路径遍历（可跳出备份目录执行任意 SQL）；`/vocab/progress` 疑似误用 `func.case`（PG 编译非法 SQL 报错）；业务路由大量内联 raw Table 而非复用 ORM；Celery/AI/支付/邮件/短信核心能力全是 TODO 占位。
- **P2**：argon2 声明但缺依赖；`AIConversation` 孤儿表；Alembic env.py 未导入 app.models（autogenerate 会丢表）；密码长度约束不一致；判分仅 `.strip().lower()` 相等。
- 亮点：管理端 RBAC 严谨、全量审计、Alembic 迁移规范（含 downgrade）、feature_flags 优雅降级、参数化查询无 SQL 注入、bcrypt 版本锁定正确。

### 3.6 services/ai-engine（FastAPI 后端）— 健康度 5/10
- **P0**：① 全站无鉴权（`/chat`/`/moderation`/`/media/*`/`/prompts` 写接口均匿名可达）；② `/media/stt` **SSRF**（服务端代抓任意 URL，可触内网/元数据）；③ `POST /prompts/{id}` 无鉴权可注入恶意 prompt 污染对话；④ 所有 provider 异常被**静默替换为"模拟响应"**——线上供应商故障/限流长期投喂假答案且不可观测。
- **P1**：同步阻塞式外部调用（在 async 路由内阻塞事件循环）；`OpenAI()` 无 timeout/max_retries；内容审核 fail-open（异常即 `flagged=False` 放行）且未被业务接口强制串联；Docker `COPY data ./data` 与 RAG `DATA_DIR` 路径不匹配 → **知识库实际加载不到**；word_games 进程内字典无锁、跨 worker 不共享（`--workers 2` 下 session 丢失）。
- **P2**：EMBEDDING_DIMENSIONS=1536 与 bge-m3(1024) 不符；requirements 含未使用 langchain/redis/pymilvus；审核 prompt `.format` 用户文本含 `{}` 会抛错被吞；全 mock 测试缺集成测试。
- 亮点：多供应商抽象层（基类→适配器→注册中心→路由）设计清晰、可扩展；离线降级分层明确；593 行 mock 测试覆盖 online/offline/empty。

### 3.7 data/ 数据质量 — 健康度 6/10
- 全部 21 个 JSON 可解析、结构合法。**引用完整性 100%**（math 5326 条、english 0 坏引用）——README "100% 引用完整性"属实；题目 id 全局唯一。
- **P0**：① 词汇规模虚标：README 称 4541+CET4，实际 kaoyan-words 仅 220、全部词汇文件合计约 520（差近 9 倍）；② 真题空壳：30 份试卷仅存结构摘要，无任何题目/答案/解析/知识点关联（与 schema 承诺脱节）；③ 习题册空壳：仅存书籍元数据+章节计数，无真实题目。
- **P1**：数学题量 5026（称 5575，缺 549），总题量 8250 未达 8300+；40 道 calculation 题缺 answer/solution；真题 schema 未覆盖 english。
- **P2**：year 字段 100% 为 null；数学知识点题目覆盖率仅 25.7%（217 个孤儿知识点）；math-full 为裸数组（建议与 english-full 对齐封装）。

### 3.8 infra / scripts / docs（部署与文档）— 一键部署不可行
- **P0（4 个阻断）**：① compose 强制要求已禁用 MinIO 的 `MINIO_*` 且 `.env.example` 缺 `POSTGRES/REDIS` 密码 → `make up` 失败；② web/mobile-web 无源码 → 构建失败；③ `Makefile`/`deploy.sh` 调用的 `scripts/seed.py`/`import_*.py`/`import_all.py` **全部不存在** → 数据初始化完全失败；④ nginx 443 引用缺省证书 → 容器启动即崩。
- **P1**：`/nginx-health` 仅定义在 443，80 健康检查永远失败；certbot 写入路径与 nginx 读取路径不一致；`api.conf` 明文 API 可达；`generate-cet4-*.ts` 写入不存在的 `server/` 目录；`check_secret` 仅前缀匹配弱密码可绕过；README 与 compose 前端清单矛盾。
- **P2**：`/static-mobile`、`/static-admin` alias 指向不存在目录；Makefile 提示端口错误；Dockerfile HEALTHCHECK 用 curl 但镜像无 curl。
- 文档一致性：旧 docs 复审报告未记录 student-web、web/mobile-web 无源码、import 脚本缺失等部署阻断项，**存在审查盲区**。

## 五、全局 P1 高危清单

- **安全**：api 默认 JWT 密钥可伪造 token、api `/system/restore` 路径遍历；ai-engine 审核 fail-open 且未串联；admin token 存 localStorage；student-web 无身份认证。
- **可用性/正确性**：ai-engine 同步阻塞外部调用、RAG 数据路径错误（知识库为空）；api 内联 raw Table、`func.case` 疑似 bug、Celery 全占位；student-web 三个答题页重复、跨科目不渲染公式；mobile 登录态不恢复、计算题判分恒错、PlanScreen 死代码；admin 裸 fetch 无 token、死链、表单数据模型错误。
- **一致性**：README 框架/前端/数据规模全面失实；docs 旧复审存在盲区；各模块类型/路由/文档三处对不齐。

## 八、总体评估

- **可用度**：当前按文档无法部署，需先修复 12 项 P0。
- **代码质量**：后端优于前端；API 管理端、AI 抽象层有真实设计功力，但前端工程配置错误普遍、移动端有启动级崩溃。
- **安全**：AI 引擎与 mobile/student-web 最薄弱（无鉴权/SSRF/无身份）；api 有密钥与路径遍历隐患。
- **数据**：引用完整性好但内容空心、规模虚标，影响产品可信度。
- **文档**：全面失实，是最大隐性风险——会持续误导后续开发者与部署者。

> 报告基于 8 个独立审查代理的逐文件实际核查，所有结论均有文件:行号依据，详见各模块审查记录。
